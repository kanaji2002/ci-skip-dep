## 研究方法

### プロジェクト選定 (PS1–PS5)

本研究では，GitHubに公開されているJavaScript/npmプロジェクトを対象に，段階的なフィルタリングを適用して分析対象を絞り込んだ。

**PS1**では，GitHub Search API（`language:JavaScript stars:>=10`）を用い，主要言語がJavaScriptであり，かつスター数が10以上のリポジトリを全件抽出した（294,477件）。

**PS2**では，2024年3月1日から2026年3月1日の調査期間内にコミット数が10件以上あるリポジトリのみを残し，継続的な開発活動が確認できるプロジェクトに絞った（46,249件）。

**PS3**では，`.github/workflows/`ディレクトリ配下にYAMLファイルが存在するリポジトリ，すなわちGitHub Actionsを採用しているプロジェクトを対象とした（22,716件）。

**PS4**では，GitHub APIを通じてActionsの実行履歴を確認し，実行記録が10件以上あるリポジトリのみを残した（18,286件）。これにより，CIが実際に稼働しているプロジェクトに限定した。

**PS5**では，GitHub APIを用いて`package.json`の存在をリポジトリごとに確認し，npmを採用しているプロジェクトのみを残した。

### RQ1: 未使用依存検知モデルの精度評価

RQ1では，「未使用依存として検知した依存関係を実際に削除してもテストが壊れないか」を基準として，各検知モデルの精度（適合率・再現率）を測定した。

#### PS6: テストスクリプトの存在確認

PS5通過リポジトリの`package.json`を取得し，`scripts.test`フィールドに実質的なテストコマンドが記述されているかを確認した。`echo "no test"` や `exit 1` などのダミー記述は除外し，テストが実際に実行可能なプロジェクトのみを残した。

#### PS7: テスト実行可能性とカバレッジ確認

PS6通過リポジトリをローカルにクローンし，`npm install`を実行後に`npm test`を走らせた。テストが通過し，かつ`nyc`（Istanbul）によるカバレッジ計測が有効なプロジェクトのみをPS7通過とした。これにより，テスト実行の信頼性が確保されたプロジェクト群を確定した。

#### 未使用依存の検知と検証

PS7のプロジェクトに対し，以下の5つのモデルを並列に実行して未使用依存を検知した。

| モデル | 手法 |
|:---|:---|
| **depcheck** | `depcheck . --json` |
| **knip** | `npx knip --reporter json` |
| **llama** | Ollama HTTP API（llama3.1:8b） |
| **qwen** | Ollama HTTP API（qwen3.5:4b） |
| **deepseek** | Ollama HTTP API（deepseek-coder:6.7b-instruct） |

各モデルが「未使用」と判定した依存パッケージを`npm uninstall`で実際に削除し，その後`npm test`を再実行して結果を記録した。判定は以下の基準による。

- **削除前PASS・削除後PASS** → 正しい検知（テストに影響なし）
- **削除前PASS・削除後FAIL** → 誤検知（実際には使用されていた）
- **削除前FAIL** → 評価不能（元からテストが壊れている）

この結果をもとに，モデルごとの適合率・再現率を算出した。

### RQ2: CIスキップ可能性の定量評価

RQ2では，未使用依存のバージョン更新によって無駄に実行されたCIビルドの割合と，それによって浪費された時間を定量化した。

まず，PS5通過プロジェクトのgit履歴を解析し，`package.json`（および`package-lock.json`）**のみ**を変更し，依存関係の単一バージョン更新だけを含むコミットを抽出した。ソースコードやREADMEなどの変更を含むコミットは除外した。

次に，抽出した各コミットについて，コミット本体と親コミットの両方を対象にGitHub Check Runs APIを呼び出し，CIの実行ステータスと実行時間を取得した。

その後，各コミットをローカルにチェックアウトして5つの検知モデルを実行し，更新された依存パッケージが未使用依存として検知されるかを確認した。

スキップ可能と判定する条件は以下のとおりである。

```
is_skippable = (更新依存 ∈ 未使用依存) AND (親コミットのビルドステータス == 'success')
```

この判定に基づき，モデルごとにスキップ可能なビルドの割合と，削減できたCIの合計時間を算出した。

### RQ3: 他言語への拡張

RQ3では，RQ2で構築したパイプラインを **Python・Rust・C#** に適用し，JavaScriptとの比較を通じて，未使用依存によるCI浪費が言語・エコシステムをまたいで共通して発生するかを検証した。

各言語について，PS1からPS5に相当する選定手順を個別に実施した。PS5の判定基準は言語ごとに対応するパッケージ管理ファイルの存在確認に置き換えた（Pythonは`requirements.txt`または`pyproject.toml`，Rustは`Cargo.toml`，C#は`.csproj`）。未使用依存の検知には，各言語向けの静的解析ツール（例：Pythonは`deptry`，Rustは`cargo-machete`）および汎用LLMモデルを適用した。スキップ判定のロジックはJavaScriptと同一である。
